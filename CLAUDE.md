## Business context

Native iOS app for a **healthy food restaurant with home delivery in Cali, Colombia**.
Operating model: dark kitchen. Main differentiator: real-time exact macronutrients per dish.

The `ios/` repository is one of several in the ecosystem. The source of truth is the Go API.
The backend OpenAPI spec drives automatic Swift type generation тАФ never write network models
by hand.

**Ecosystem stack (for context only тАФ do not touch from this repo):**
- Backend: Go + PostgreSQL + Redis
- Web: SvelteKit
- Android: Jetpack Compose
- iOS: SwiftUI (this repo)

## Architecture тАФ Non-negotiable rules

### Base pattern: MVVM + Ports & Adapters

```
View (.swift)
  тФФтФАтФА ViewModel (@Observable)
        тФФтФАтФА Repository (Protocol = port)
              тФФтФАтФА RepositoryLive (struct = real adapter)
              тФФтФАтФА RepositoryMock (struct = test adapter)
                    тФФтФАтФА APIClient (generated by swift-openapi-generator)
```

**Never** call the generated client directly from a View or ViewModel.
**Always** go through a repository Protocol.

### Dependency injection: @Environment, never singletons

```swift
// тЬЕ Correct тАФ injected from above, testable
@Environment(OrdersViewModel.self) var viewModel

// тЭМ Forbidden тАФ coupled, not testable
OrdersViewModel.shared
APIClient.shared
```

Real dependency wiring happens **only** in `FoodAppApp.swift` (the entry point).
Think of it as the `<Providers>` wrapper in React. No other file constructs concrete dependencies.

## Code conventions

### Naming тАФ use the business glossary

The official glossary is at `playbook/docs/09-GLOSARIO.md`. Swift code uses the **English**
terms exactly as defined there. Never invent your own translations.

| Spanish (business) | Swift (code) |
|---|---|
| ├Нtem de men├║ | `MenuItem` |
| Pedido | `Order` |
| L├нnea de pedido | `OrderItem` |
| Receta | `Recipe` |
| Ingrediente | `Ingredient` |
| Estado del pedido | `OrderStatus` |
| Canal | `OrderChannel` |

**Order status** (required enum):
`pending` тЖТ `confirmed` тЖТ `preparing` тЖТ `ready` тЖТ `dispatched` тЖТ `delivered` / `cancelled`

Never use string literals for statuses. Always use the enum.

### Required suffixes

| Type | Suffix | Example |
|---|---|---|
| Protocol (port) | `Protocol` | `MenuRepositoryProtocol` |
| Real implementation | `Live` | `MenuRepositoryLive` |
| Mock implementation | `Mock` | `MenuRepositoryMock` |
| ViewModel | `ViewModel` | `MenuViewModel` |
| Feature main view | `View` | `MenuView` |

### Modern Swift тАФ minimum requirements

- **iOS 17+ minimum.** Use `@Observable` (not `ObservableObject`/`@Published`).
- **async/await always.** Zero callbacks, zero Combine for networking.
- **Structured concurrency.** Use `.task {}` in Views, `async let` for parallelism.
- **Value types preferred.** `struct` for models and repositories. `class` only for ViewModels.
- **No force unwrap (`!`).** Handle optionals with `guard let`, `if let`, or `??`.

## HTTP client тАФ Generator rules

The HTTP client is **never written by hand**. It is generated from `openapi.yaml` using
`swift-openapi-generator`.

```yaml
# openapi-generator-config.yaml
generate:
  - types
  - client
```

When the backend changes its API:
1. Update `openapi.yaml`
2. `Cmd+B` in Xcode тАФ the client regenerates automatically
3. The compiler will tell you exactly what broke

**Generated client wrapper** (only place where the base URL is set):

```swift
// Core/Networking/GeneratedClient+Setup.swift
import OpenAPIURLSession

extension Client {
    static func live() -> Client {
        Client(
            serverURL: URL(string: "https://api.yourdomain.co")!,
            transport: URLSessionTransport()
        )
    }
}
```

## Repositories тАФ Standard structure

Each repository defines the Protocol, Live, and Mock in a single file.
Follow **exactly** this pattern for all features:

```swift
// Features/Menu/MenuRepository.swift

// MARK: - Protocol (port)
protocol MenuRepositoryProtocol {
    func getMenuItems() async throws -> [Components.Schemas.MenuItem]
    func getMenuItemDetail(id: String) async throws -> Components.Schemas.MenuItem
}

// MARK: - Live (real adapter)
struct MenuRepositoryLive: MenuRepositoryProtocol {
    private let client: Client

    init(client: Client) {
        self.client = client
    }

    func getMenuItems() async throws -> [Components.Schemas.MenuItem] {
        let response = try await client.getMenuItems()
        return try response.ok.body.json
    }

    func getMenuItemDetail(id: String) async throws -> Components.Schemas.MenuItem {
        let response = try await client.getMenuItemById(.init(path: .init(id: id)))
        return try response.ok.body.json
    }
}

// MARK: - Mock (for tests and Previews)
struct MenuRepositoryMock: MenuRepositoryProtocol {
    func getMenuItems() async throws -> [Components.Schemas.MenuItem] {
        return [
            .init(id: "1", name: "Protein bowl", calories: 520, protein: 42, carbs: 35, fat: 18),
            .init(id: "2", name: "Mediterranean salad", calories: 380, protein: 28, carbs: 22, fat: 14)
        ]
    }

    func getMenuItemDetail(id: String) async throws -> Components.Schemas.MenuItem {
        try await getMenuItems().first!
    }
}
```

## ViewModels тАФ Standard structure

```swift
// Features/Menu/MenuViewModel.swift
import Foundation

@Observable
final class MenuViewModel {
    // MARK: - State (equivalent to useState in React)
    var menuItems: [Components.Schemas.MenuItem] = []
    var isLoading = false
    var errorMessage: String? = nil

    // MARK: - Dependency (always the Protocol, never the Live)
    private let repository: any MenuRepositoryProtocol

    init(repository: any MenuRepositoryProtocol) {
        self.repository = repository
    }

    // MARK: - Actions
    func loadMenuItems() async {
        isLoading = true
        errorMessage = nil
        do {
            menuItems = try await repository.getMenuItems()
        } catch {
            errorMessage = "Could not load the menu. Please try again."
        }
        isLoading = false
    }
}
```

## Views тАФ Standard structure

```swift
// Features/Menu/MenuView.swift
import SwiftUI

struct MenuView: View {
    // Consumes the injected ViewModel тАФ never constructs it
    @Environment(MenuViewModel.self) private var viewModel

    var body: some View {
        Group {
            if viewModel.isLoading {
                ProgressView()
            } else if let error = viewModel.errorMessage {
                ErrorView(message: error) {
                    Task { await viewModel.loadMenuItems() }
                }
            } else {
                menuList
            }
        }
        .task {
            // Like useEffect(() => {}, []) in React
            await viewModel.loadMenuItems()
        }
        .navigationTitle("Menu")
    }

    private var menuList: some View {
        List(viewModel.menuItems, id: \.id) { item in
            MenuItemRow(item: item)
        }
    }
}

// MARK: - Preview (always with Mock, never with real data)
#Preview {
    NavigationStack {
        MenuView()
            .environment(
                MenuViewModel(repository: MenuRepositoryMock())
            )
    }
}
```

**View rules:**
- Views are **dumb** тАФ they only display state and delegate actions to the ViewModel.
- Private subviews with `private var` to keep `body` readable.
- **Always** include a `#Preview` with the Mock. No Preview means the code is incomplete.
- Break long Views into subcomponents in the same file (if small) or separate files within
  the same feature.

## Dependency wiring (entry point)

```swift
// FoodAppApp.swift
import SwiftUI

@main
struct FoodAppApp: App {
    // Build the real dependency graph here
    // Like <QueryClientProvider><AuthProvider> in React

    @State private var menuVM: MenuViewModel
    @State private var orderVM: OrderViewModel
    @State private var authVM: AuthViewModel

    init() {
        let client = Client.live()
        _menuVM = State(initialValue: MenuViewModel(
            repository: MenuRepositoryLive(client: client)
        ))
        _orderVM = State(initialValue: OrderViewModel(
            repository: OrderRepositoryLive(client: client)
        ))
        _authVM = State(initialValue: AuthViewModel(
            repository: AuthRepositoryLive(client: client)
        ))
    }

    var body: some Scene {
        WindowGroup {
            ContentView()
                .environment(menuVM)
                .environment(orderVM)
                .environment(authVM)
        }
    }
}
```

## Business domain тАФ What the app does

### Features by priority (aligned with roadmap T4)

**App MVP (launch):**
1. **Catalog / Menu** тАФ list items with macros (calories, protein, carbs, fat)
2. **Item detail** тАФ full macros, ingredients, possible substitutions, allergens
3. **Cart** тАФ add/remove items, view total, per-item notes
4. **Checkout** тАФ select address, apply promo code, initiate Wompi payment
5. **Auth** тАФ sign up, login, Sign in with Apple (required by App Store), Google
6. **Order tracking** тАФ real-time status: `pending` тЖТ `delivered`
7. **Profile** тАФ saved addresses, order history, notifications

**Post-launch (month 3+):**
- Nutritional personalization (goals: weight loss, muscle gain, maintenance)
- HealthKit тАФ log order macros to Apple Health (T12)
- Weekly subscriptions (T10)
- Referral program (T11)

### Nutritional data тАФ The differentiator

Macros come from the backend (calculated by the recipe engine).
The app **never calculates macros locally** тАФ it only displays them.
`macros_snapshot` in `order_items` is the snapshot at order time тАФ use it
for history, do not recalculate.

### Order status тАФ State machine

Strictly follow this flow for notifications and UI:

```
pending тЖТ confirmed тЖТ preparing тЖТ ready тЖТ dispatched тЖТ delivered
                                                      тЖШ cancelled
```

Each status transition **may trigger a push notification** (APNs).
The user-facing message per status is defined in the business glossary.

## Payments тАФ Wompi

- Amounts always in **COP cents** (├Ч100). `$25,000` = `2500000`.
- **Never store** card data. Only the Wompi transaction reference.
- Flow: app creates order тЖТ backend creates Wompi transaction тЖТ app opens Wompi WebView/redirect
  тЖТ webhook to backend confirms payment тЖТ backend updates status to `confirmed`
  тЖТ app receives update via polling or WebSocket.

---

## Push notifications (APNs)

- Use the `UserNotifications` framework.
- Device token is sent to the backend on login (stored in `user_sessions`).
- Notifications are **informational about order status**, not invasive marketing.
- Request notification permission **after** the first order is placed, not on first app launch
  (Apple rejects apps that request permissions without context).

## Security тАФ Rules

- JWT and refresh tokens: store in **Keychain**, never in UserDefaults.
- Never log tokens, payment data, or personal information.
- HTTPS always. Do not disable certificate validation even in development.
- Comply with Colombian Law 1581 of 2012 (personal data) тАФ show privacy policy in onboarding.

## Testing тАФ Rules

### What always needs unit tests:
- ViewModels (state logic, error handling, edge cases)
- Data transformations (COP price formatting, statuses, macros)

### What does not need unit tests:
- Views (validated with Previews)
- Mocks (test helpers, not business logic)

### ViewModel test pattern:

```swift
// Features/Menu/MenuViewModelTests.swift
import Testing
@testable import FoodApp

@Suite("MenuViewModel")
struct MenuViewModelTests {

    @Test("loads items successfully")
    func loadMenuItemsSuccess() async {
        let vm = MenuViewModel(repository: MenuRepositoryMock())
        await vm.loadMenuItems()

        #expect(vm.menuItems.isEmpty == false)
        #expect(vm.isLoading == false)
        #expect(vm.errorMessage == nil)
    }

    @Test("handles network error correctly")
    func loadMenuItemsFailure() async {
        let vm = MenuViewModel(repository: MenuRepositoryFailingMock())
        await vm.loadMenuItems()

        #expect(vm.menuItems.isEmpty == true)
        #expect(vm.errorMessage != nil)
    }
}
```

Use the **Swift Testing** framework (`import Testing`) тАФ it is the modern one, not XCTest.

## Errors тАФ Standard handling

```swift
// Core/AppError.swift
enum AppError: LocalizedError {
    case networkUnavailable
    case unauthorized          // 401 тАФ redirect to login
    case notFound              // 404
    case serverError           // 5xx
    case unknown(Error)

    var errorDescription: String? {
        switch self {
        case .networkUnavailable: return "No connection. Check your internet."
        case .unauthorized:       return "Your session expired. Please sign in again."
        case .notFound:           return "We couldn't find what you're looking for."
        case .serverError:        return "Server error. Try again in a few minutes."
        case .unknown:            return "Something went wrong. Please try again."
        }
    }
}
```

ViewModels convert errors from the generated client to `AppError` before exposing them to the View.
Views display `errorMessage` (String), never the raw technical error.

## What NOT to do тАФ Explicit list

```
тЭМ Singletons (APIClient.shared, ViewModel.shared)
тЭМ Force unwrap (!) in production code
тЭМ Business logic inside a View
тЭМ Hand-written network models (the generator handles that)
тЭМ String literals for order statuses ("pending", "confirmed"...)
тЭМ Calculating macros on the client (backend's responsibility)
тЭМ Storing card data or sensitive payment information
тЭМ ObservableObject / @Published (use @Observable)
тЭМ Combine for networking (use async/await)
тЭМ JWT tokens in UserDefaults (use Keychain)
тЭМ HTTP calls without error handling
тЭМ Preview without mock data
тЭМ Terms not found in glossary 09-GLOSARIO.md
```
